# CLAUDE.md

Context for AI agents (and future-you) working on this repo. Read this fully
before making changes — it captures decisions and gotchas that aren't obvious
from the code alone. See the "Keeping this file current" section at the
bottom for the rule on updating it.

## What this is

**GoldTrack** — a local-first web app for tracking Digital Gold investments.
No backend, no accounts, no server-side storage of user data. Everything the
user enters lives in a `.gtrack` file they explicitly save/open. A free
GitHub Actions cron checks the public spot gold price daily, keeps a running
history, computes a simple BUY/SELL/HOLD heuristic, and can notify the user
on Telegram.

Live site: **https://invest.shammas.in** (custom domain on GitHub Pages).
Repo: `mshammas/invest`.

## Tech stack (all free-tier, by design)

- Vite + React + TypeScript + Tailwind v4, in `app/`
- Hosting: GitHub Pages, deployed via GitHub Actions (not the legacy
  branch-based Pages source)
- Gold spot price: [gold-api.com](https://api.gold-api.com/price/XAU) — free,
  keyless, no rate limit on the real-time endpoint
- Historical backfill: [freegoldapi.com](https://freegoldapi.com/) — free,
  keyless, World Bank monthly data (1960–2024) + Yahoo Finance daily
  (2025–present)
- Alerts: Telegram Bot API (user creates their own bot via BotFather)
- FX rates for currency conversion:
  [frankfurter.dev](https://api.frankfurter.dev/v1/latest?from=USD) — free,
  keyless, CORS-open, ECB-sourced daily rates
- No paid services anywhere in the stack. Keep it that way unless the user
  explicitly asks to add a paid dependency.

## Key decisions and why (read before "fixing" these)

- **`vite.config.ts` has `base: '/'`.** The site is served from the root of
  the custom domain `invest.shammas.in`, not a GitHub Pages project-page
  subpath. If the custom domain is ever removed, this would need to change
  back to `/invest/` (its original value) to match `mshammas.github.io/invest/`.

- **`index.css` forces `color-scheme: light`.** Without it, some browsers
  auto-invert an unstyled page (no explicit background/text color) into a
  garish dark rendering. The whole app is intentionally light-only right now —
  there is no dark mode implementation, just this pin.

- **The gold-signal heuristic uses a *calendar-day* window, not "last N
  entries."** `data/gold-price-history.json` contains decades of backfilled
  history (much of it monthly, from `freegoldapi.com`) plus real daily entries
  from the cron. If the signal logic used "last 90 entries," a backfilled
  dataset with a gap right before "today" would make the app think it has
  plenty of recent data and compute a misleading BUY/SELL comparison against a
  stale price. Instead, `computeSignal()` filters to entries whose *date*
  falls within the last `TRAILING_WINDOW_DAYS` (90) of the actual run date, and
  gates `MIN_HISTORY_DAYS` (14) on that same real window. Never change this
  back to an index-based slice.

- **`gold-signal.yml`'s commits do NOT trigger `deploy.yml` via `on: push`.**
  `stefanzweifel/git-auto-commit-action` pushes using the default
  `GITHUB_TOKEN`, and GitHub deliberately blocks `GITHUB_TOKEN`-authored
  pushes from triggering other workflows (anti-infinite-loop safeguard). This
  caused an 11-day silent staleness bug where the daily cron kept updating the
  data in the repo but the live site never redeployed. Fixed by adding a
  `workflow_run` trigger to `deploy.yml` keyed to `gold-signal.yml`'s
  completion. If you ever see the live site out of sync with what's in
  `data/gold-price-history.json` on `origin/main`, check this trigger chain
  first before assuming the cron itself is broken.

- **Git remote uses an SSH host alias, not plain `github.com`.** This
  machine has two GitHub accounts configured in `~/.ssh/config`
  (`github-mshammas` and `github-ichummas`). The origin remote is
  `git@github-mshammas:mshammas/invest.git` — using plain `github.com` risks
  picking up whichever key is cached in the SSH agent (which caused a push to
  fail with a permission error to the wrong account earlier). Don't
  "simplify" the remote URL back to the plain host.

- **Transaction entry is quantity (g/mg) + total amount, not price/gram.**
  Digital gold purchases are naturally described as "₹2000 for 129mg," not as
  a per-gram rate. `pricePerGram` is derived (`totalAmount / grams`), stored
  on the transaction for display, but never a required input.

- **Price history chart is hand-rolled SVG, no charting library.** Built
  following the `dataviz` skill's method (single series → no legend needed;
  2px line, round caps; calendar-time x-scale, not index-based; hover
  crosshair + tooltip). If you touch `PriceHistorySection.tsx`, re-read that
  skill first rather than improvising chart conventions. There used to be a
  daily-rate table below the chart making every value reachable without
  hovering (the skill's usual recommendation) — removed by explicit user
  request in favor of the chart + hover tooltip alone; don't re-add it
  without checking first.

- **UI is deliberately "elder-friendly."** Base font size bumped ~12%
  (`html { font-size: 112.5% }` in `index.css`), buttons/inputs have a
  44–48px minimum touch target, borders are 2px (not 1px) for contrast, and
  the transaction list is a labeled-card layout rather than a dense table.
  Keep new UI consistent with this — don't introduce small touch targets or
  low-contrast text.

- **The market-price display currency (`CurrencyPicker`) is separate from a
  portfolio's `settings.currency`.** The latter is what a user's own
  transactions are recorded in (set at file creation, used by
  `Dashboard`/`TransactionForm`/`TransactionList`); the former only controls
  what currency `SignalBanner` and `PriceHistorySection` convert the USD/oz
  spot price into for display, and is a standalone `localStorage` preference
  (`goldtrack-display-currency`) independent of any open `.gtrack` file. Don't
  conflate the two or wire the picker into `PortfolioFile.settings`.

- **FX rates are fetched client-side from frankfurter.dev and cached in
  `localStorage` for the day** (`goldtrack-fx-rates`, keyed by date) — see
  `fetchExchangeRates()` in `currency.ts`. On fetch failure it falls back to
  the last cached rates, and if there's no cache at all, to `{ USD: 1 }`
  (everything reads as USD numbers until a fetch succeeds). Frankfurter only
  covers ECB-tracked currencies, which is why `CURRENCIES` in `currency.ts` is
  the major-currency set + INR rather than every possible currency — adding a
  currency not on Frankfurter's list will silently show USD-equivalent values
  (rate falls back to 1) since `convertFromUsd()` has no way to know it's
  missing.

- **No gh CLI auth in the agent sandbox.** Cannot query GitHub Actions run
  status, secrets, or Pages settings directly via `gh api`/`gh run list`.
  When debugging deploy/workflow issues, either ask the user to check the
  Actions tab, or inspect state indirectly via `git fetch` + `git log
  origin/main` + `git show origin/main:<path>`.

## Current scope (v1, built)

- Transaction tracking (buy/sell) with weighted-avg cost, invested amount,
  realized P&L, and optional manual current-price for unrealized P&L
- New/Open/Save/Save As via File System Access API with fallback
- Daily BUY/SELL/HOLD/ACCUMULATING_HISTORY signal + Telegram alerts on change
- Price history chart (30D/90D/1Y/5Y/All) with hover tooltip, a currency
  picker (major currencies + INR), and a unit toggle (troy oz / gram /
  8 grams) that convert the USD/oz spot price for display only
- IndexedDB crash-recovery autosave

## Deferred / roadmap (not built — see README.md's Roadmap section too)

- Password-based encryption of the exported file
- CSV export
- XIRR and FIFO capital-gains lot tracking
- PWA/offline support
- User-editable alert thresholds (currently fixed: 5% dip → BUY, within 1% of
  high → SELL)
- Multiple portfolio files / goals
- Vendor-specific pricing (the displayed rate is still spot gold only —
  vendors charge a markup this app doesn't model)

## Conventions

- No code comments unless explaining a non-obvious *why* (a gotcha, a
  workaround, a subtle invariant) — never restate *what* the code does.
- Minimal dependencies. Before adding a library, check whether a small
  hand-rolled solution is reasonable (this is why the chart has no charting
  library, and the cron script has zero npm dependencies).
- Everything free-tier. Don't introduce a service that requires payment or a
  metered API key beyond a generous free tier without asking the user first.
- Commit messages explain *why*, not just *what*.

## Keeping this file current

**Update this file whenever a change would leave it stale or misleading** —
not just when asked. That includes (non-exhaustive):
- A new architectural decision or a decision reversed/changed
- A new gotcha discovered (something that broke non-obviously and cost time
  to debug)
- A new external service/dependency added or removed
- The directory structure changing shape
- Scope moving from "deferred" to "built," or a new deferred item identified
- Any workaround whose reason isn't obvious from the code

Keep entries concise — this file is a map and a gotcha-list, not a changelog.
If something here becomes wrong (not just outdated), fix or remove it rather
than layering a correction on top.
